The Workflow as a Pipeline

Before any single skill, see the shape of the whole. Two kinds of skill, one loop — and your first install.

You already drive Claude Code hard. The thing mattpocock/skills changes isn't power — it's discipline. Matt's thesis: an agent amplifies whatever habits you give it. Feed it vague requirements and skipped tests, and it produces vague, untested code faster. Feed it a tight loop — align, spec, slice, test, tend — and it compounds good engineering. — mattpocock/skills; Claude Code for Real Engineers

So the first thing to learn is not a skill. It's the map: how the skills chain, and the one distinction that governs all of them.

The one distinction: who pulls the trigger

Every skill is either user-invoked or model-invoked. Get this and the whole collection snaps into focus.

User-invoked You type /skill-name. These are deliberate moves: grilling yourself for clarity, writing a PRD, slicing issues, reviewing architecture. You decide when.
Model-invoked Claude fires them itself when the situation matches the skill's description. You set them up once; then writing a feature auto-engages tdd, and hitting a bug auto-engages diagnosing-bugs. You don't ask — the discipline is just on.

This is why installing the skills changes behaviour even when you never type a command: the model-invoked ones quietly raise the floor on everything the agent does.

The loop, end to end

A feature travels this pipeline. Terracotta tops = you invoke; teal tops = the model invokes mid-work. The full annotated version lives in your Skill Map reference — keep that open.

01 ALIGNgrill-with-docsInterview → domain model in CONTEXT.md
02 SPECto-prdConversation → PRD in tracker
03 SLICEto-issuesPRD → vertical-slice issues
04 BUILDtddRed → green → refactor
05 FIXdiagnosing-bugsReproduce → minimize → fix
06 TENDimprove-codebase-architecturePeriodic design review

Notice the order is non-negotiable for a reason: alignment before code. The grilling and PRD stages exist to kill ambiguity before the agent writes anything — because an agent that misunderstands you will build the wrong thing flawlessly, fast.

Your win: install the skills on the anchor repo

Tangible win By the end of this step the full pipeline is live in your real monorepo — ~/code/example-app — and the model-invoked skills are already working, no commands needed.

Run these in the anchor repo (your teacher can run them with you — just ask):

# in ~/code/example-app
npx skills@latest add mattpocock/skills    # pick the skills (start: take all)
/setup-matt-pocock-skills                   # wire up tracker, triage labels, docs location

The setup step is itself a user-invoked skill — it asks where your issues live and where docs go. Answer for this repo (it already has a docs/ folder and uses git). When it finishes, you've completed stage 0 of the loop. Install command per the repo README.

Watch for You have a lot installed already (GSD, SEO suites, MCP). mattpocock/skills is a separate namespace and won't collide — but after install, run /ask-matt once to confirm the router sees them. If a name clashes with an existing skill, tell your teacher and we'll note it.
