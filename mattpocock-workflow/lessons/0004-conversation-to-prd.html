Lesson 4 · Spec

Conversation → PRD

Grilling resolved the decisions. Now freeze them into one durable document the work can derive from.

By the end of Lesson 2 you had grilled a feature until every decision branch was resolved, and in Lesson 3 you sharpened the domain model behind it. But a conversation is volatile — it scrolls away, it lives in one agent session, it cannot be handed to a teammate or a fresh context. The next move turns that hard-won clarity into something durable: a PRD.

What to-prd does

to-prd is user-invoked: you type it when a feature has been talked through. It synthesizes the conversation into a single Product Requirements Document and publishes it to the tracker you configured during /setup-matt-pocock-skills. — mattpocock/skills

User-invoked You decide the feature is ready to crystallize, then type /to-prd. It does not run on its own — a PRD is a deliberate commitment, not something the agent should snapshot mid-thought.

PRD vs CONTEXT.md — do not confuse them

This is the trap. Both are documents you produce with the skills, but they answer different questions:

DocumentScopeLifespan
CONTEXT.mdThe standing domain model — vocabulary, entities, rules for the whole project.Lives and grows with the repo.
PRDThe intent of one feature — what to build and why, this time.Created per feature; issues derive from it.

A PRD uses the vocabulary that CONTEXT.md establishes — which is exactly why grilling and domain modeling come first. A PRD written against a sharp domain model is terse and unambiguous; one written against a vague one is padded with re-explanation.

Your win: produce a real PRD for example-app

Tangible win Turn a grilled feature conversation into a real PRD on ~/code/example-app — for example a new capability for the example-service (you already have service-spec.md and roadmap.md to ground it). When done, you have a spec that Lesson 5 can slice into issues.

In the anchor repo, after talking a feature through with the agent:

# in ~/code/example-app
# 1. grill the feature first (Lesson 2) so decisions are resolved
# 2. then crystallize it:
/to-prd

If the skill asks where to publish, point it at your tracker or the repo docs/ folder. Read the PRD it produces and check: does every requirement use vocabulary your CONTEXT.md already defines? If it has to re-explain a term, that term belongs back in CONTEXT.md. Behavior per the to-prd skill.

🧹 Clear here for best performance The PRD is now published to your tracker, so /clear now. A short, focused context is where an LLM does its sharpest work — less to attend to means tighter reasoning, less drift onto the long grilling transcript, and lower cost and latency. The slice stage (/to-issues) reads the PRD directly, so it needs none of that chat history. Full map of clear points: the runbook.
