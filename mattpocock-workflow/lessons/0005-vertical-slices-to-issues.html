Vertical Slices: PRD → Issues

A PRD is one big thing. Work happens in small things. How you cut it decides everything that follows.

You have a PRD from Lesson 4. Now it has to become issues an agent (or a teammate) can pick up and finish. The single most consequential decision here is how you cut. Cut it wrong and every later stage — TDD, review, handoff — inherits the mess. to-issues exists to cut it the right way: into vertical slices. — mattpocock/skills

The key idea: vertical, not horizontal

Imagine the PRD is a layered cake — UI on top, business logic in the middle, data at the bottom. There are two ways to cut it.

CutWhat each issue containsCost
Vertical sliceOne thin capability, end to end: a bit of UI + its logic + its data.Ships value immediately; grabbable in parallel; testable on its own.
Horizontal layerAll the data, then later all the logic, then later all the UI.Nothing works until the last layer lands; blocks parallel work; value is delayed.

Concrete example on a product site: a "show related products" feature. The vertical cut is one issue that takes a single product, computes one related item, and renders it — end to end, demoable. The horizontal cut is "build the recommendations table", then "build the API", then "build the component" — three issues where you cannot see anything work until all three are done.

User-invoked to-issues is a move you make: you type it against a finished spec. It breaks the PRD into issues that are independently grabbable — each one a vertical slice someone can take without waiting on the others.

Then route them: triage

Once the slices exist, triage (also user-invoked) moves issues through a state machine of triage roles — so each issue lands in the right state instead of sitting in an undifferentiated pile. — mattpocock/skills

Your win: slice a real PRD on example-app

Tangible win Take the PRD from Lesson 4 and turn it into a set of vertical-slice issues for ~/code/example-app, then triage them. When done, you have at least one issue that is small, end-to-end, and ready for the TDD loop in Lesson 6. 
# in ~/code/example-app
/to-issues   # slice the PRD into vertical-slice issues
/triage      # route each issue into the right state

Sanity check the output: pick any one issue and ask "could I demo this on its own?" If the answer is no, it is probably a horizontal layer — tell the agent to re-slice it end to end. Behavior per the to-issues skill.

🧹 Clear here for best performance The slice issues are in your tracker now, so /clear — then build each issue in its own fresh window. A short, focused context is where an LLM does its sharpest work — one issue in view means tighter reasoning, less drift, and lower cost and latency. Each build reads its issue (plus CONTEXT.md) on its own. Full map of clear points: the runbook.
