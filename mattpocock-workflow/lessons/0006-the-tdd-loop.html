The TDD Loop

A failing test first. Then the smallest code that passes. Then cleanup — with the test holding the line.

You have a sliced issue (Lesson 5) that already carries the intent of the PRD. Now the agent builds it — and this is where most AI coding quietly goes wrong: the model writes plausible code, you skim it, and nothing ever proves it does the right thing. The tdd skill closes that gap by making a test the first artifact, not an afterthought. — mattpocock/skills

Red → green → refactor

The cycle is three small moves, repeated tightly. The win of each move is so small it fits in working memory — which is exactly why it works with an agent moving fast.

REDwrite a testEncode the intent of the issue as a test that fails for the right reason
GREENmake it passThe smallest change that turns the test green — no more
REFACTORclean upImprove the design with the test as a safety net

The test is not bureaucracy — it is the feedback loop. Red proves the test can fail (so a pass means something). Green keeps the change minimal (so you do not over-build). Refactor is safe precisely because the test will scream if you break behaviour.

Model-invoked tdd is fired by Claude, not by you. Once installed, when the agent sets out to implement a feature it engages this loop on its own — so the discipline is simply on. Your job is to give it a sharp issue and to notice if it skips straight to code without a failing test first.

Why the test comes from the issue

Because you grilled (Lesson 2) and sliced (Lesson 5), the issue states one thin end-to-end behaviour. That behaviour is the test: "given X, the function returns Y." TDD then turns your earlier alignment work into an executable check. Skip the grilling and the test just encodes a guess.

Check yourself

Recall before you reveal — the effort is what makes it stick.

Your win: ship one slice test-first

Tangible win Take a single vertical-slice issue from Lesson 5 and watch it go red → green → refactor on the real repo — a tested unit in example-service/ that proves it does what the issue says.

In ~/code/example-app, point the agent at one issue and insist on the failing test before any implementation:

# in ~/code/example-app
# pick the smallest slice from your issue list, then:
"Implement issue #<n> test-first. Write the failing test for the
 behaviour first, run it red, then the minimal code to go green,
 then refactor. Do not write implementation before the test."

pnpm test            # the loop's feedback — run it red, then green

If the agent jumps to implementation, stop it and ask for the red test — enforcing that order is the skill. tdd is model-invoked; see it in the repo.

🧹 Clear here for best performance The slice is green and committed, so /clear before the next issue. A short, focused context is where an LLM does its sharpest work — carrying one finished slice into the next only adds noise that means more drift and higher cost. The committed tests and code are the record; the next issue starts clean from its own issue text. Full map of clear points: the runbook.
