Every skill, who fires it, and where it sits in the loop. The compressed essence — return here, not to the lessons.

Source of truth: github.com/mattpocock/skills. The repo evolves — re-check exact names and descriptions against it. This map is synthesized from the repo and Matt's Claude Code for Real Engineers workshop.

The core distinction

Every skill is one of two kinds. Internalizing this is half the discipline:

KindWho triggers itHow you use it
USER-INVOKED You, explicitly You type /skill-name (or ask for it). These are moves you choose to make — planning, triage, setup.
MODEL-INVOKED Claude, automatically Claude reads the description and fires it when the situation matches (writing a feature → tdd; a bug appears → diagnosing-bugs). You set them up; the agent reaches for them.

The pipeline at a glance

A feature flows left to right. Terracotta = you invoke; teal = the model invokes during work.

01 ALIGNgrill-with-docsInterview to build the domain model → CONTEXT.md
02 SPECto-prdTurn the conversation into a PRD in your tracker
03 SLICEto-issuesBreak the PRD into vertical-slice issues
04 BUILDtddRed → green → refactor on each slice
05 FIXdiagnosing-bugsReproduce → minimize → hypothesize → fix
06 TENDimprove-codebase-architecturePeriodic design review against entropy

Running underneath all of it: git-guardrails-claude-code blocks dangerous git commands, and setup-pre-commit wires lint/type/test gates so the loop can't regress silently.

Glossary

Terms used consistently across every lesson. When a lesson uses one of these, it means exactly this.

TermMeaning
SkillA single SKILL.md file (YAML frontmatter + body) that defines a repeatable workflow the agent follows.
User-invokedA skill you trigger explicitly with /name. A deliberate move.
Model-invokedA skill Claude triggers itself when the situation matches its description.
CONTEXT.mdThe project's living domain model — shared vocabulary that makes agent sessions terse and aligned.
GrillingA relentless interview loop that forces every ambiguous decision to be resolved before code is written.
PRDProduct Requirements Document — the synthesized intent, published to your tracker, that issues derive from.
Vertical sliceAn issue that delivers one thin end-to-end piece of value, independently grabbable — not a horizontal layer.
Red-green-refactorTDD cycle: write a failing test (red), make it pass (green), then clean up (refactor).
Deep module(Ousterhout) A module with a small interface hiding a lot of implementation — the goal of codebase-design.
Progressive disclosureLoading short skill summaries first, full bodies only when needed — the v1.0 token-saving design.
